# 任务五变更内容：统一 Bot 渠道命令派发逻辑

## 变更目标
根据重构优化计划，`telegram.ts` 与 `feishu.ts` 中存在高度重复的命令解析、AI 结果响应格式化和 Isolate 挂起超时保护逻辑。本任务的目标是将这些重复的核心流程提取为共享模块，以保证行为一致性并降低维护成本。

## 详细变更记录

### 1. 新增共享模块文件
在 `src/worker/handlers/` 目录下新增了共享模块：
- **`src/worker/handlers/bot-shared.ts`**
  - 定义并导出了统一的命令解析器 `parseCommand`，集中处理 `/command`, `/command@bot`, `/command args` 等格式的解析。
  - 定义了通用的响应结果提炼函数 `extractResult`，将原先在各渠道分散判断的 AI `data` 参数抽象为类型安全的 `FormattedResult` 结构。
  - 提供了超时检查逻辑 `isResponseExpired` 统一处理由于 Isolate 休眠引发的幽灵消息拦截。

### 2. 改造 Bot Handler 逻辑
针对 Telegram 与飞书 对应的接收处理器执行了依赖注入与精简：

- **`src/worker/handlers/telegram.ts`**
  - 移除了独立的命令解析正则表达式，替换为 `parseCommand` 并实现了逻辑衔接。
  - 重构了 `formatTelegramResponse`，现在仅负责依据 `extractResult` 返回的基础字段添加 Telegram 特有的 HTML 标签与格式样式，解耦了业务取值逻辑。
  - 使用 `isResponseExpired` 替代了内联的超时丢弃判断。

- **`src/worker/handlers/feishu.ts`**
  - 同步移获得了各自为战的正则表达式解析，改用共享 `parseCommand` 接口。
  - 修改 `formatFeishuResponse`，基于公共返回结构实现飞书支持的纯文本排版。
  - 将内联超时检查更新为 `isResponseExpired`。

## 验证与结论
- 已通过 `npx tsc --noEmit` 保证 TypeScript 静态类型重构未出现属性遗漏。
- 已执行自动化的 API 测试用例回归，均成功通过。
- 架构调整保持了原有平台的鉴权与事件去重（如飞书）的独立性，未引入预期外的副作用。
