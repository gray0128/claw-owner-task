# 任务六变更内容：推送通知逻辑收敛

## 变更目标
根据重构优化计划，`summary.ts` 和 `remind.ts` 中存在重复的多渠道（Bark、Telegram、飞书）推送分发逻辑。本任务的目标是将这些逻辑提取为统一配置的 `notify` 服务，减少代码冗余并确保各渠道通知行为的一致性。

## 详细变更记录

### 1. 基础服务增强
- **`src/worker/services/bark.ts`**
  - 为 `sendBarkNotification` 函数新增了可选的 `linkUrl` 参数。
  - 实现了将链接自动附加到 Bark 推送地址的 `?url=` 查询参数中，确保用户点击通知时可以跳转到指定页面（如任务总结页）。

### 2. 新增多渠道推送服务
- **`src/worker/services/notify.ts`**
  - 新建了统一的推送分发中心。
  - 导出了 `sendToAllChannels` 异步函数，该函数会根据环境变量自动识别并向已配置的渠道（Bark、Telegram、飞书）发送通知。
  - 抽象了 `NotifyOptions` 接口，支持传递标题、纯文本内容、HTML 内容及跳转链接，自动处理不同渠道的格式差异（如 Telegram 的 HTML 逃逸，飞书的 token 获取及消息重试）。
  - 支持 `source` 过滤逻辑，防止向消息触发源重复推送通知。

### 3. 处理器逻辑重构
- **`src/worker/handlers/summary.ts`**
  - 大幅精简了 AI 总结生成后的推送代码，由原先约 50 行的碎片化逻辑缩减为单次 `sendToAllChannels` 调用。
  - 移除了冗余的服务导入（如 `getTenantAccessToken`, `sendFeishuMessage`），改由 `notify` 服务内部统一处理。
  - 保持了原有的 Bark 跳转链接特性及针对不同触发源的推送过滤规则。

## 验证与结论
- **类型检查**: 已通过 `npx tsc --noEmit` 验证，确保新增服务与现有代码的类型完全兼容。
- **功能回归**: 已执行全量 API 自动化测试，所有 24 个用例均成功通过，证明重构未对核心业务造成影响。
- **代码质量**: 显著降低了代码重复率，提高了推送逻辑的可维护性。
