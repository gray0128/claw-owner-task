# 接入飞书机器人需求说明

## 1. 背景与目标
当前 **AI优先的任务管理系统 (claw-owner-task)** 已经成功接入了 Telegram 和 QQ Bot 作为对话式交互入口（Chatbots）。为了进一步丰富多平台支持，满足更多国内企业或团队用户的日常办公场景需求，计划将 **飞书（Lark）机器人** 作为新的 Chatbot 接入系统。

由于系统后端基于 Cloudflare Workers 构建，且已具备成熟的 Webhook 路由分发和 AI 语义解析能力，接入飞书应用机器人（Webhook 模式）在架构上具有极高的契合度。

## 2. 功能需求
1. **事件订阅与消息接收**：
   - 提供公网 Webhook URL 供飞书开放平台订阅事件。
   - 支持飞书的 URL 验证（Challenge）机制。
   - 接收并解析来自飞书用户的单聊或群聊消息事件（`im.message.receive_v1`）。
2. **AI 意图解析与任务操作**：
   - 将提取出的纯文本消息转发至现有的 `aiHandlers` 进行语义解析。
   - 支持通过自然语言对话实现任务的添加、查询、总结等核心功能。
3. **消息回复与展示**：
   - 调用飞书 OpenAPI (`im/v1/messages`) 将任务操作结果回传给对应的会话 (`chat_id` 或 `open_id`)。
   - 针对不同类型的操作结果进行格式化（MVP 阶段使用纯文本拼接，后续可演进为飞书消息卡片 Interactive Message Card 以提升视觉体验）。
4. **鉴权与安全性**：
   - 校验飞书回调请求的 Signature，确保请求来源的合法性。
   - （可选）支持通过 `FEISHU_ALLOWED_CHAT_ID` 等环境变量限制仅允许特定的群组或用户与机器人交互。

## 3. 技术架构与设计

### 3.1 接入模式
采用 **企业自建应用 + Webhook 事件订阅模式**：
- **触发链路**：用户在飞书发送消息 -> 飞书开放平台 POST JSON 到 Worker Webhook -> Worker 处理并响应 200 -> Worker 内部调用 `fetch` 主动请求飞书 API 回复消息。

### 3.2 环境变量配置
需要在 `wrangler.toml` (留空) 和 `.dev.vars` 中新增以下环境变量：
- `FEISHU_APP_ID`: 飞书应用的 App ID。
- `FEISHU_APP_SECRET`: 飞书应用的 App Secret（用于获取 tenant_access_token）。
- `FEISHU_ENCRYPT_KEY`: 事件订阅的 Encrypt Key（若开启加密，需用于 AES 解密）。
- `FEISHU_VERIFY_TOKEN`: 事件订阅的 Verification Token（用于签名校验）。
- `FEISHU_ALLOWED_CHAT_ID`: （可选）限制交互的 Chat ID 列表，以逗号分隔。

### 3.3 核心模块规划
- **服务层 (`src/worker/services/feishu.ts`)**：
  - `getTenantAccessToken(appId, appSecret)`: 获取并在内存中短效缓存 `tenant_access_token`。
  - `sendFeishuMessage(token, receiveId, msgType, content)`: 封装发送消息 API 的 HTTP 调用。
  - `verifyFeishuSignature(timestamp, nonce, encryptKey, body, signature)`: 实现飞书 Webhook 请求的安全验签（SHA256）。
  - `decryptFeishuEvent(encryptKey, encryptStr)`: 实现 AES-256-CBC 事件解密（如果飞书后台开启了加密）。
- **路由层 (`src/worker/handlers/feishu.ts`)**：
  - 处理 POST 请求，判断是否存在 `challenge` 字段，若有则直接返回响应以通过配置验证。
  - 提取事件中的 `event.message.content`，解析内部 JSON 获取真实文本。
  - 调用 `aiHandlers.request` 调度 AI 任务处理流程。
  - 根据返回结果，调用 `sendFeishuMessage` 回复用户。
- **入口集成 (`src/worker/index.ts`)**：
  - 新增路由：`app.route('/api/webhook/feishu', feishuHandlers)`。

## 4. 接口交互流程
1. **飞书后台配置校验**：
   飞书 POST `{"challenge": "xxx", "type": "url_verification"}` -> Worker 响应 `{"challenge": "xxx"}`。
2. **正常消息处理链路**：
   用户发消息 -> Worker 收到事件 -> 验签并（或）解密 -> 提取 `open_id`/`chat_id` 和文本 -> AI 分析生成操作结果 -> Worker 异步通过 `fetch` 调用 `https://open.feishu.cn/open-apis/im/v1/messages` -> 返回成功响应给用户所在的飞书会话。

## 5. 安全与限制
- **Token 时效性**：飞书 `tenant_access_token` 有效期 2 小时，请求频率上限较高。初期可在每次回复前实时获取（低频场景下），后续若有高频需求，需引入缓存机制（内存变量或 Cloudflare KV）。
- **请求限流**：应用机器人接口调用限流为 1000次/分钟、50次/秒，系统内部 `fetch` 需确保不对飞书网关发起恶意高并发请求。
- **消息长度限制**：飞书纯文本消息体积上限为 150KB，常规任务数据返回完全满足。

## 6. 实施步骤
1. **基础设施配置**：在飞书开放平台创建应用，申请发送/接收消息权限，并获取凭证。配置本地/线上环境的环境变量。
2. **基础 Service 编写**：实现 `src/worker/services/feishu.ts` 中的解密、验签和 Token 获取方法。
3. **Webhook 接口开发**：完成 `src/worker/handlers/feishu.ts` 路由的编写与 AI 处理流的桥接。
4. **系统集成与发布**：在 `index.ts` 注册路由，使用 `ngrok` 或 `cloudflared` 进行本地联调，测试通过后部署至生产环境。

## 7. 具体实现方案记录

### 7.1 核心服务层开发 (`src/worker/services/feishu.ts`)：
- `getTenantAccessToken`: 实现了获取并调用飞书 API 授权凭证的方法。
- `sendFeishuMessage`: 封装了调用 `im/v1/messages` 向单聊/群聊发送任务操作结果的功能。
- `verifyFeishuSignature`: 实现了 `X-Lark-Signature` 的 SHA-256 签名校验，以保证 Webhook 来源的安全性。
- `decryptFeishuEvent`: 实现了 AES-256-CBC 的消息体解密逻辑，以防后续在飞书后台开启事件加密。

### 7.2 Webhook 路由层开发 (`src/worker/handlers/feishu.ts`)：
- 暴露 `/api/webhook/feishu` 接口并处理 POST 请求。
- 自动响应飞书开放平台应用验证时的 Challenge，**始终返回 HTTP 200 及 `{"code": 0}` JSON 格式以防飞书网关判定超时重试**。
- 解析 `im.message.receive_v1` 用户发给机器人的单聊或群聊消息，通过内置规则清除消息中包含的 `@机器人` 文本，确保意图识别的纯净。
- 将文本透传至现有的 `aiHandlers` 与 `taskHandlers`，并对 `/add` 和 `/summary` 等特殊指令实现了快速分发。
- 对 AI 返回的结果进行文字格式化，复用现存分享链接及列表卡片的逻辑回传给飞书应用内对应的话话。

### 7.3 路由入口与环境配置 (`src/worker/index.ts` & `wrangler.toml`)：
- 在 `index.ts` 注册了 `app.route('/api/webhook/feishu', feishuHandlers)` 路由，并在 `Bindings` 中声明了相关环境字段。
- 在 `src/worker/middleware/auth.ts` 中，将飞书 Webhook 路由 (`/api/webhook/feishu`) **加入全局鉴权白名单**，交由路由内部独立处理 `X-Lark-Signature` 验签。
- 在 `wrangler.toml` 文件中留出了对应飞书凭据的环境变量配置块：
```toml
# FEISHU_APP_ID = "YOUR_FEISHU_APP_ID"
# FEISHU_APP_SECRET = "YOUR_FEISHU_APP_SECRET"
# FEISHU_VERIFY_TOKEN = "YOUR_VERIFY_TOKEN"
# FEISHU_ENCRYPT_KEY = "YOUR_ENCRYPT_KEY"
# FEISHU_ALLOWED_CHAT_ID = "YOUR_ALLOWED_CHAT_ID"
```

## 8. 完整配置与发布流程指南

1. **创建飞书应用**：在飞书开放平台创建企业自建应用，并启用“机器人”能力。
2. **配置 Webhook URL**：
   - 进入“事件订阅”，填写形如 `https://your-domain.com/api/webhook/feishu` 的请求地址。
   - 飞书会发送包含 `Challenge` 的验证请求，Worker 需返回 200 OK 且包含 Challenge 的 JSON 进行确认。
3. **订阅事件与权限申请**：
   - 在“事件订阅”中添加 `im.message.receive_v1`（接收消息）事件。
   - 在“权限管理”中申请 `im:message`（发消息）及 `im:message.p2p_msg:readonly` / `im:message.group_at_msg:readonly`（读消息）权限。
4. **注入 Cloudflare 环境变量**：
   - 从飞书后台“凭证与基础信息”和“事件订阅”页面获取相关的 ID 和 Key。
   - 通过 `npx wrangler secret put <KEY_NAME>` 依次注入 `FEISHU_APP_ID`、`FEISHU_APP_SECRET`、`FEISHU_VERIFY_TOKEN`、`FEISHU_ENCRYPT_KEY` 以及目标用户的 `FEISHU_ALLOWED_CHAT_ID`。
5. **发布上线**：在飞书后台“版本管理与发布”中创建一个新版本并申请发布，审核通过后即可在飞书客户端内与机器人交互。
