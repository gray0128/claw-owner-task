# 语音消息统一处理架构与模块设计文档

本文档详细说明了在 `claw-owner-task` 中如何统一支持飞书和 Telegram 语音消息的处理。

## 1. 架构设计目标

- **统一入口抽象**：针对不同平台（飞书/Telegram）的语音接收差异，提供一致的转译调用接口。
- **免除外部存储**：利用 Cloudflare Worker 的边缘计算和流式传输能力，通过代理路由解决公网访问鉴权问题，无需额外集成 R2/S3。
- **内聚与高复用**：将 ASR (自动语音识别) 逻辑提取为独立的服务模块，方便未来新增其他平台的语音输入（如微信/企业微信）。
- **异步与健壮性**：将语音识别这类耗时操作纳入 `waitUntil` 生命周期，确保 Webhook 及时响应，并完善异常容错回退机制（转译失败不影响系统继续运行，可通过文字提示用户）。

## 2. 核心模块划分

整个语音处理功能将被划分为以下几个核心模块：

### 2.1 ASR 服务模块 (`src/worker/services/asr.ts`)
负责与火山引擎 Seed-ASR 2.0 接口直接通信。
- `submitAsrTask(audioUrl: string)`: 发送请求到火山引擎，提交给定公网 URL 的音频转译任务，并返回 `requestId`。
- `pollAsrResult(requestId: string)`: 轮询查询 `requestId` 的结果，返回转译完毕的纯文本文字。
- `processAudioToText(audioUrl: string)`: 组合封装提交和轮询的完整逻辑，供上层调用。

### 2.2 飞书适配与代理模块 (`src/worker/handlers/audio-proxy.ts` 及 `src/worker/services/feishu.ts`)
解决飞书语音文件的鉴权下载问题。
- **服务层 (`services/feishu.ts`)**: 新增 `getFeishuResourceStream(messageId, fileKey)` 获取飞书音频的原始二进制流。
- **路由层 (`handlers/audio-proxy.ts`)**: 新增专门供火山引擎回调拉取的端点。
   - `GET /api/proxy/feishu-audio` 
   - 接收参数：`message_id`, `file_key`, `sign` (签名防盗链)。
   - 功能：验证签名后，调用 `getFeishuResourceStream`，并直接把 Response Body (Stream) Pipe 回发请求的客户端。

### 2.3 Telegram 适配层 (`src/worker/services/telegram.ts`)
解决 Telegram `file_id` 到公网下载 URL 的转换。
- `getTelegramVoiceUrl(fileId: string)`: 调用 Telegram `getFile` API 获取文件 Path，拼接为可直接公网下载的 URL（由于自带 Token 鉴权，无需经过 `audio-proxy` 转发）。

### 2.4 平台事件处理器改造 (`src/worker/handlers/feishu.ts` & `telegram.ts`)
在现有的 Webhook 处理器中侵入逻辑，统合上述模块。

## 3. 详细处理时序流转

### 飞书处理时序：
1. `feishu.ts` 收到 `message_type === 'audio'` 事件。
2. 从 `content` 解析出 `file_key` 和 `message_id`。
3. 生成一个带签名的 Worker 内部代理下载 URL：`https://<worker-domain>/api/proxy/feishu-audio?message_id=xxx&file_key=yyy&sign=hash`。
4. 调用 `asr.ts` 的 `processAudioToText(代理URL)`。
5. 等待转译文字返回后，拼装为 `[语音转写]: xxx` 传入原本的 AI 处理流程 (`aiHandlers.request`)。

### Telegram 处理时序：
1. `telegram.ts` 收到 `message.voice` 事件。
2. 从 `voice` 对象中提取 `file_id`。
3. 调用 `telegram.ts` 的 `getTelegramVoiceUrl(file_id)`，得到形如 `https://api.telegram.org/file/bot<token>/<file_path>` 的原生公网 URL。
4. 调用 `asr.ts` 的 `processAudioToText(Telegram原生URL)`。
5. 等待转译文字返回后，拼装为 `[语音转写]: xxx` 传入原本的 AI 处理流程 (`aiHandlers.request`)。

## 4. 关键代码接口设计草案

### 4.1 `asr.ts` (统一 ASR 抽象)
```typescript
export interface AsrConfig {
    apiKey: string;
}

export async function processAudioToText(audioUrl: string, config: AsrConfig): Promise<string> {
    // 1. Submit task
    const requestId = await submitTask(audioUrl, config);
    // 2. Poll until success/timeout
    return await pollResult(requestId, config);
}
```

### 4.2 `audio-proxy.ts` (飞书公网代理出口)
```typescript
app.get('/feishu-audio', async (c) => {
    const messageId = c.req.query('message_id');
    const fileKey = c.req.query('file_key');
    const sign = c.req.query('sign');
    
    // 1. 验证签名 (利用 env.TASK_API_KEY 作为盐值计算 HMAC/MD5)
    if (!verifySign(messageId, fileKey, sign, c.env.TASK_API_KEY)) {
        return c.text('Forbidden: Invalid Signature', 403);
    }
    
    // 2. 请求飞书获取流 (透传 header 和 body)
    const token = await getTenantAccessToken(...);
    const feishuRes = await fetchFeishuResource(token, messageId, fileKey);
    
    // 3. 原样返回流
    return new Response(feishuRes.body, {
        headers: { 'Content-Type': 'audio/ogg' }
    });
});
```

## 5. 开发任务清单与计划

按照合理的依赖关系，建议的开发步骤如下：

1. **环境配置更新**：在 `.dev.vars` 和文档中新增 `VOLC_API_KEY` 环境变量说明。
2. **ASR 服务开发**：新建并开发 `src/worker/services/asr.ts` 模块。
3. **代理路由开发**：新建 `src/worker/handlers/audio-proxy.ts`，实现防盗链鉴权与飞书音频流转发，并在主入口注册路由。
4. **飞书 Handler 支持**：修改飞书代码，增加 `audio` 类型支持，串联代理生成与 ASR。
5. **Telegram Handler 支持**：修改 Telegram 代码，增加 `getFile` 获取 URL 逻辑，串联 ASR。
6. **自动化及边缘边界测试**：编写相关单测及集成测试场景模拟（包括静音/识别错误导致的回退文字提示）。
