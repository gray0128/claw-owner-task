# 飞书语音消息接入系统分析报告

基于 `docs/volcengine-asr` 项目的参考以及当前 `claw-owner-task` 的代码架构，分析得出：**本项目完全支持将飞书语音消息转换为文本并接入 AI 处理流程，且拥有较高的可行性。**

## 1. 核心流程与原理

1. **接收语音事件**：
   目前的 `src/worker/handlers/feishu.ts` 已经处理了 `message_type === 'text'`。我们可以扩展此逻辑，同时捕获 `message_type === 'audio'` 的消息事件。
2. **提取语音文件信息**：
   飞书语音消息的 `content` 是一个 JSON 字符串，包含 `file_key` 和相关的音频属性（通常是 ogg/opus 格式）。
3. **获取语音文件**：
   通过飞书 API（`GET /open-apis/im/v1/messages/{message_id}/resources/{file_key}?type=audio`），携带现有的 Tenant Access Token (`getTenantAccessToken`) 下载音频数据。
4. **提交 ASR 转换任务**：
   火山引擎 Seed-ASR 2.0 API（如参考项目 `volcengine.js` 所示）要求传递一个**公开的 HTTP(S) URL** 供其拉取音频。
5. **获取识别文本并喂给 AI**：
   使用火山的轮询接口等待转换完成。获取纯文本后，可附加上述前缀（例如：`[语音转译]: ...`），直接复用现有 `aiHandlers.request` 逻辑。

## 2. 核心技术痛点与解决方案

最主要的挑战点在于：**火山引擎 ASR 只能读取公开 URL，而飞书的获取资源 API 是带鉴权的，无法直接提供给火山引擎 ASR。**

针对这个问题，在 Cloudflare Workers 环境下，有两个可行的解决方案：

### 方案一：Cloudflare Worker 内部代理转流（推荐）
在 API 项目内部新增一个拉取音频的中转路由：
- **原理**：创建一个类似于 `GET /api/feishu/proxy-audio?id=xxx&key=yyy&token=zzz` 的公开端点。
- **流程**：火山引擎请求该 URL 时，Worker 后端带上飞书的鉴权 Header，去请求飞书源站音频流，然后直接 **Pipe 流转发** 回给火山引擎。
- **优势**：纯内存流转，不需要引入任何外部存储服务（如 R2、S3），延迟极低，系统架构保持简单！
- **注意事项**：需要在参数中带有一次性或带签名的 `token` 以防恶意拉取。

### 方案二：集成 Cloudflare R2（与参阅项目类似）
- **原理**：仿照 `docs/volcengine-asr/scripts/s3-client.js`，引入 R2（或者 S3 兼容的存储）。先把音频下载到 ArrayBuffer 中，然后上传到 R2 并暴露公网临时链接。
- **优势**：标准、稳定，适合同时还需保存相关语音记录的系统。
- **劣势**：需要额外去配置 Cloudflare R2 存储桶，配置较繁琐。

## 3. 编码要求匹配性验证
- **格式免转码**：飞书应用发来的语音通常是 `ogg/opus` 编码格式。火山引擎大模型录音版 (Seed-ASR) **原生支持 `format: "ogg", codec: "opus"`**。这在 Worker 环境下是巨大的优势，因为 Worker 环境无法运行 `ffmpeg`。
- **超时机制**：当前代码处理 AI 流程已被包含在 `c.executionCtx.waitUntil(...)` 异步执行上下文中，直接返回了 `code: 0` 给飞书 Webhook 以防超时。而语音转录（往往数秒到十几秒）可以安全地放入这个 waitUntil 的异步生命周期内执行，完美契合现有架构。

## 4. 后续开发实施建议

1. 安装 `axios` 或复用现有 `fetch` 封装火山引擎的鉴权与发包逻辑。
2. 扩写 `src/worker/services/feishu.ts` 中的飞书接口，新增一个通过 `file_key` 和 `message_id` 拉取语音二进制数据的函数。
3. 采用**方案一（内部路由流转）**，在 Worker 新开一个 `proxy-audio` 子路由专供火山下载。
4. 修改 `src/worker/handlers/feishu.ts`，增加处理 `audio` 类型的逻辑结构。


## 5. Telegram 语音消息接入分析

基于对 Telegram Bot API 的调研，**Telegram 同样且更容易支持语音消息接入**。

### 核心差异与优势

1. **音频格式直接兼容**：Telegram 的语音消息 (`message.voice`) 默认格式与飞书一致，通常也是 OGG/Opus，这完美契合了火山引擎 Seed-ASR 2.0 的要求免去了转码烦恼。
2. **下载免额外鉴权，直接可得 URL**：与飞书复杂的由于 Token 鉴权下载不同，在 Telegram API 中：
   - 首先通过 `getFile` API (传入 `voice.file_id`) 获取文件路径 (`file_path`)。
   - 然后直接拼接成 URL: `https://api.telegram.org/file/bot<token>/<file_path>`。
   - **重大优势**：这个 URL 虽然包含 bot token 且需要保密，但它是**可以直接被公开访问的 HTTP GET 请求**（不需要在 Header 里带额外鉴权参数）。
   - 这意味着，我们**完全不需要**为 Telegram 开发类似飞书的 `proxy-audio` 中转路由，可以直接将这个包含 token 的 Telegram 文件 URL 原样传给火山引擎 ASR！

### 实施建议

1. 扩展 `src/worker/handlers/telegram.ts`，在现有的 `body?.message?.text` 基础之上，增加对 `body?.message?.voice` 的捕获拦截。
2. 提取 `file_id`，调用一把 Telegram `getFile` 获取路径，生成下载 URL。
3. （可选）为安全起见，如果不希望火山引擎直接摸到我们的 bot token URI，也可以一并走上一节提到的 `proxy-audio` 中立转发路由清洗掉 URL，但在技术实现上直接传是可行的。
4. 获取转录文本后统一组装 prompt 投递给 AI (`aiHandlers.request`)。结论：**可行性极高且实现复杂度低于飞书**。
