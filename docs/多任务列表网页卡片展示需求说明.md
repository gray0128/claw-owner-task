# 需求说明：聊天渠道多任务列表卡片展示

## 1. 背景与目标
当前系统在与聊天机器人（Chatbots，如 Telegram）交互时，如果 AI 解析后返回的是多条任务的列表数据（如：查询所有待办任务），系统会将任务列表直接拼接成气泡长文本发送给用户。
这种方式存在以下问题：
1. **气泡过长**：当任务较多时，聊天窗口内的文本过长，影响聊天体验，容易造成“刷屏”。
2. **信息不全**：为了避免气泡过长，当前格式被大幅简化，无法在聊天窗口中完整展示每个任务的详细信息（如长描述、创建时间、标签等）。

为了解决上述问题，计划引入“**多任务列表网页卡片**”机制：当用户请求返回多个任务时，系统不再返回长文本，而是生成一个独立的、完整的网页，并在聊天渠道中通过**包含链接及其预览卡片**的形式进行优雅展示。

## 2. 核心交互流转

1. **用户发起请求**：用户通过 Telegram 发送类似“查询明天的所有任务”的指令。
2. **后端 AI 解析与执行**：
   - AI 成功解析并调用查询接口。
   - 后端获取到包含多条任务记录的数组（列表数据）。
3. **网页生成与持久化**：
   - 如果数据为**列表（多条）**，系统自动为其生成一个独立的网页凭证（UUID），并关联该批任务数据。
4. **聊天渠道响应**：
   - 后端仅向聊天渠道返回一段极其精简的文案。
   - **格式规范**：`意图描述（一句话） + 换行 + 网页链接`。
   - *示例*：
     ```text
     ✅ 成功查询到 5 个明天的任务。
     https://<domain>/list/:uuid
     ```
   - *注意*：系统需确保 Telegram 请求未禁用链接预览（已在前序修复中移除 `disable_web_page_preview: true`），以便 Telegram 自动展开该链接的富文本预览卡片。

## 3. 详细设计规格

### 3.1 数据库结构与生命周期
复用或借鉴现有的“单条任务独立查看 (Task Share)”和“AI 总结网页 (Task Summaries)”的生命周期管理机制：
*   **存储机制**：需新建数据表或利用现有表（如可扩展 `task_shares` 表以支持列表数据，或新建 `task_lists` 表）存储 UUID、关联的原始查询参数或直接存储查询结果的快照。
    *   *建议策略*：直接将任务列表数据（或其核心参数的快照）以 JSON 格式存储于数据库中，类似 `task_summaries`。
*   **访问时效**：与 AI 总结网页一致，链接有效期为 **24 小时**。
*   **状态处理**：过期后再次访问链接时，展示优雅的过期提示：“该任务列表已过期，请重新发起查询”。后台利用定期清理脚本（Cron）对过期数据进行物理删除。

### 3.2 列表网页模板设计 (UI/UX)
为了承载可能较多的数据量，该模板需要全新设计，侧重于信息密度与浏览效率：
*   **设计原则**：极简、现代、高信息密度。**网页模板必须被设计为可迭代优化的架构，页面的 UI/UX 将随着使用反馈进行持续迭代。**
*   **布局结构**：
    *   **头部**：展示本次查询的核心意图/标题（如“明天的任务列表”）、总数及生成时间。
    *   **列表区**：任务以卡片或紧凑列表形式排列。
*   **卡片内容完整性**：
    *   必须包含：标题（Title）、优先级标识（Color/Icon）、状态（Status）、截止/提醒时间（Dates）。
    *   支持展示：长描述（Description，过长可提供展开/折叠或适度截断）、所属分类（Category）、关联标签（Tags）。
*   **移动端优先**：确保在手机端（特别是 Telegram 应用内浏览器）拥有优秀的阅读体验（如合理的字体大小、行高和间距）。

### 3.3 接口与逻辑变更
*   **Web 渲染路由**：新增公共路由 `GET /list/:uuid`，读取数据并渲染上述 HTML 模板。
*   **响应拦截层**：
    *   在 `src/worker/handlers/telegram.ts` 中拦截 AI 处理成功返回的列表数据。
    *   当且仅当数据为数组且长度 `> 0` 时，触发 UUID 生成、数据持久化及消息格式重组。

## 4. 边界与异常处理
*   **空列表场景**：当查询结果为 0 条时，**不生成网页**。直接在聊天窗口回复简短提示：“没有找到符合条件的任务。”
*   **单条结果场景**：当查询结果虽然为数组，但实际仅包含 **1 条**数据时，**系统必须复用《单条任务独立查看需求说明》中的逻辑**。即，为其生成一个单条任务的 `/share/:uuid` 链接，并在聊天渠道中以此链接和对应的意图描述进行响应，不再使用多任务列表视图。

## 5. 验收标准
1.  **交互**：在 Telegram 中查询多个任务时，收到的是一句话加链接，并且能够自动弹出网页卡片预览。
2.  **视觉**：点击链接打开的新页面布局整洁，在手机和 PC 端均不出现横向滚动条，任务要素显示完整。
3.  **时效**：生成的链接在 24 小时后精确失效并显示对应的过期提示页面。

## 6. Web 任务管理主界面卡片化改造说明

> 对应 GitHub Issue #13：用户希望将当前任务管理界面由过长的纵向列表改为卡片式，并根据网页宽度自适应展示。此处描述的是 `src/web/index.html` 进入登录态后的主任务管理页，不是 Telegram 查询结果的 `/list/:uuid` 临时分享页。

### 6.1 改造目标

1. **降低列表纵向长度**：将 `#tasksContainer` 从单列长列表升级为响应式卡片网格，在桌面宽屏下并排展示多张任务卡片。
2. **保持移动端可读性**：小屏设备仍使用单列卡片，卡片内容不能被挤压、重叠或产生横向滚动。
3. **保留现有功能**：搜索、状态筛选、优先级筛选、分类筛选、标签点击筛选、编辑、删除、完成等交互不改变语义和事件入口。
4. **延续当前视觉语言**：继续使用现有低饱和色、细边框、克制排版和抽屉/弹窗交互，不引入新构建流程或前端依赖。

### 6.2 页面布局规格

#### 容器宽度

- `.app-screen` 最大宽度应从当前偏窄的单列阅读宽度提升到适合卡片网格的管理台宽度，建议使用 `max-width: 1180px` 或接近值。
- 页面左右内边距使用 `clamp()` 或媒体查询自适应，保证桌面端有足够呼吸空间，移动端不浪费宽度。

#### 任务网格

- `#tasksContainer.tasks-list` 改为 CSS Grid：
  - 桌面端：`grid-template-columns: repeat(auto-fill, minmax(280px, 1fr))`。
  - 中等宽度：根据可用宽度自动变为 2 列。
  - 移动端：低于约 `680px` 时回到单列。
- 网格间距建议保持在 `12px-16px`，避免任务密集时阅读负担过高。
- 空状态和加载状态需要横跨全部网格列，可通过 `.empty-state, .loading-state { grid-column: 1 / -1; }` 处理。

#### 任务卡片

- 现有 `.task-row` 应转为卡片样式，而不是表格行样式：
  - 使用 `display: flex; flex-direction: column;` 或单卡片内部 grid。
  - 使用 `border: 1px solid var(--border)`、`background: var(--surface)`、`border-radius: var(--radius)`。
  - `min-height` 保持稳定，避免按钮出现或描述长度变化导致整体跳动明显。
  - `completed` 状态继续降低透明度并保留标题删除线。
- 卡片内部建议分区：
  - 顶部：标题、优先级点、状态。
  - 中部：描述，最多显示 4-6 行，超出用渐隐或截断，后续可扩展展开/折叠。
  - 底部：截止时间、提醒时间、分类、标签。
  - 操作区：编辑、完成、删除等操作始终可见或在桌面 hover 显示、移动端常显。

### 6.3 信息展示规则

- 标题必须换行并使用 `overflow-wrap: anywhere`，避免长任务名撑破卡片。
- 描述使用 `white-space: pre-wrap` 保留换行，但应设置合理的最大行数，避免单张卡片过长。
- 标签继续使用当前 `.tag-link` 交互，点击后写入 `activeTagFilter` 并刷新筛选状态。
- 状态、优先级、时间等元信息在卡片内使用紧凑 chip 或短文本，保证任务要素无需打开编辑弹窗也能快速扫描。

### 6.4 响应式验收断点

- **桌面宽屏（>= 1024px）**：至少可展示 3 列任务卡片，页面不再呈现一条很长的单列列表。
- **平板/窄桌面（680px-1023px）**：自动展示 2 列或根据容器宽度自适应。
- **手机（< 680px）**：单列卡片，筛选工具栏和头部操作不重叠，任务操作按钮可直接点击。

### 6.5 验证步骤

1. 启动本地开发服务器：`npm run dev`。
2. 使用测试账号或 API Key 登录 Web 页面。
3. 准备至少 12 条不同状态、优先级、分类和标签的任务数据。
4. 分别在约 `390px`、`768px`、`1280px` 视口下检查：
   - 任务是否按 1/2/3+ 列自适应。
   - 页面是否无横向滚动。
   - 长标题、长描述、多个标签是否不会撑破卡片。
   - 搜索和筛选后的空状态是否仍居中且横跨网格。
5. 回归点击操作：
   - 新建任务。
   - 编辑任务。
   - 标记完成。
   - 删除任务。
   - 点击标签筛选并清除筛选。

### 6.6 接受标准

1. 登录后的任务管理页在桌面宽屏下展示响应式卡片网格，而不是单列长列表。
2. 移动端保持单列卡片并完整保留原有任务操作。
3. 所有筛选、搜索、标签过滤、编辑、完成、删除功能保持可用。
4. 未引入新的构建步骤、第三方运行时依赖或后端 API 变更。
5. 视觉风格与现有 Web 界面一致，并通过浏览器视口检查确认无明显重叠、溢出和横向滚动。
