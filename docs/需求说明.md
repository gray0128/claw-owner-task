# AI优先的任务管理系统需求说明 (AI-Enhanced & Cross-Platform)

## 项目概述

这是一个专为个人用户和 **AI Agent (OpenCLaw)** 设计的任务管理工具。提供云端存储、跨平台 CLI 以及多渠道提醒能力，强调对 AI 的透明度与操作的极简性。

## 核心设计原则

### 1. AI 友好性 (AI-First)
- **上下文溯源**: 记录任务来源 (`source`) 及其背后的原始上下文 (`metadata`)。
- **Schema 自发现**: AI 可通过 `/api/info` 接口动态获取当前系统的所有配置枚举。
- **语义化搜索**: 支持对任务标题和描述的模糊匹配，方便 AI 快速查找任务。
- **Telegram 对话式交互**: Telegram 机器人不仅作为推送通道，同时也是 AI 操作的入口。用户在 Telegram 发送的消息等同于本地执行 `claw-task ai "输入内容"`，云端 AI 会自动解析并处理任务。处理结果及提醒消息都将经过格式排版优化，以适应聊天窗口的展示习惯。

### 2. 跨平台与提醒解耦 (Channel Separation)
- **跨平台 CLI**: 基于 Node.js，支持 macOS, Linux, Windows/WSL。不依赖特定 OS 脚本。
- **多路提醒**:
    - **云端渠道 (Bark / Telegram)**: 
        - 由 Cloudflare Worker Cron 触发推送。
        - 支持并发推送至 iOS Bark App 和 Telegram 机器人。
        - **Telegram 专属优化**: 针对输出到 Telegram 的消息，进行 Markdown 渲染和排版优化，以适合聊天窗浏览。
    - **本地渠道 (OpenCLaw)**: 由本地 Cron 触发 CLI，交由 OpenCLaw 自身频道执行。
- **逻辑**: 提醒系统完全独立于任务的 `completed` 流程，通过 `remind_at` 触发，并在提醒完成后**自动将其推送到下一个有效未来时间**（周期任务）或**设为 NULL**（单次任务），防止冗余推送。

### 3. 功能克制 (Less is More)
- **无嵌套任务**: 不支持子任务。推荐在 `description` 中使用 Markdown 检查单 (`- [ ]`)。
- **轻量级重复**: 支持简单的周期规则 (`daily`, `weekly`, `monthly`)。

## 核心配置

- **`TASK_API_KEY`**: (Secret/Env) 强制 Bearer Token 校验。
- **`USER_TIMEZONE`**: (Env) 默认 `Asia/Shanghai` (UTC+8)。用于解析不带时区标识的日期时间字符串。
- **`BARK_URL`**: (Secret) 仅供云端 Worker 推送使用。
- **`TELEGRAM_BOT_TOKEN` & `TELEGRAM_CHAT_ID`**: (Secret) 用于 Telegram 提醒下发及与之相关的双向 AI 会话（可选）。

## 时间处理规范 (Timezone & UTC)

本系统采用 **“以服务端配置为基准”** 的时间处理原则，确保跨平台用户体验的一致性：
1. **统一存储**: 所有时间字段在数据库中均以标准的 ISO UTC 格式存储。
2. **强制标准化**: 后端 API 在接收到日期时间字符串时，会根据 **服务端 Worker 的 `USER_TIMEZONE` 配置** 执行逻辑：
    - **优先级**: 显式时区标识 (如 `+08:00`) > 服务端 `USER_TIMEZONE` > 缺省北京时间 (`Asia/Shanghai`)。
    - **逻辑**: 对于不带时区的字符串，系统自动按上述优先级补齐时区偏移量，并强制转换为 UTC 存入。
3. **显示逻辑**: 
    - **Web 端**: 自动调用 `/api/info` 获取服务端的 `USER_TIMEZONE` 并通过 `Intl.DateTimeFormat` 渲染时间。即：无论用户所在地理位置，看到的展示时间均与服务端配置的时区一致。
    - **CLI 端**: 目前展示原始存储的 UTC 字符串（建议开发者在集成时根据业务需求自行转换）。
4. **一致性保证**: 当本地 CLI 配置的时区与服务端 Worker 不同时，只要输入不带时区的字符串，系统将始终以服务端配置为准进行校准，确保任务提醒的触发时间（由云端 Cron 判定）符合预期。

## 功能需求

### 1. 任务管理 (CRUD+)
- **状态机**: `pending`, `in_progress`, `completed`, `cancelled`。
- **计算属性**: `overdue` (当状态为 `pending` 且 `due_date` 已过时由 API 判定)。
- **重复逻辑**: 任务完成时，若有 `recurring_rule`，则自动按周期推算下次时间并重置为 `pending`。

### 2. 分类与标签管理
- **分类 (Category)**: 物理归类，一对一。支持 CRUD。
- **标签 (Tag)**: 逻辑标记，多对多。支持 CRUD 及级联删除。
  - **格式约束**: 标签名称仅允许包含中文字符、英文字母及数字，**禁止包含任何特殊字符或空格**。
  - **易用性增强**: API 及 CLI 支持“按名称使用”。传递标签时支持中英文逗号混合分隔（如 `tags="紧急，工作, P1"`）。对于不存在的合法标签名称，系统将自动创建并关联，无需预先获取 ID。

### 3. CLI 功能
- **全局命令名**: `claw-task` (避免与业界通用的 `task` 冲突)。
- **全局参数**: `--json` 用于输出机器友好的 JSON 格式，替代默认的表格排版（当发生错误时也会抛出标准 JSON）。
- **基本命令**: 
  - 任务: `add` (支持 ISO 和 YYYY-MM-DD 时间格式，执行严格校验), `list` (输出含 Tags 和 Remind 列，支持按状态、优先级、分类、标签名称、到期时间及提醒时间筛选), `update` (支持修改已有任务), `complete`, `delete` (提供容错退路)。
  - 分类与标签: `tags`, `add-tag`, `categories`, `add-category`。
  - 帮助: `--help` 能够无副作用地展示帮助文档，不强制依赖 API Key 环境变量。
- **提醒检查**: `check --channel agent` (由本地 Cron 调用)。
- **Bark 推送日志**: 在云端提醒触发成功后完成日志记录，自动清理 7 天前的历史记录。
- **离线感知**: 在 API 不可用时输出清晰的离线提示。

## 数据模型

### Task 表 (核心)
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键 |
| title | TEXT | 任务标题 |
| description | TEXT | 支持 Markdown 检查单 |
| status | TEXT | `pending/in_progress/completed/cancelled` (默认 `pending`) |
| priority | TEXT | `low/medium/high` (默认 `medium`) |
| category_id | INTEGER | 关联分类 ID (`ON DELETE SET NULL`) |
| source | TEXT | 来源，如 `user`, `openclaw` (默认 `user`) |
| metadata | TEXT | JSON 格式存储 AI 上下文 |
| recurring_rule | TEXT | `daily/weekly/monthly/none` (默认 `none`) |
| due_date | DATETIME | 截止时间 (UTC 存储) |
| remind_at | DATETIME | 提醒时间 (UTC 存储) |

| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

### Bark_logs 表 (推送审计)
| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键 |
| task_id | INTEGER | 关联任务 ID (`ON DELETE SET NULL`) |
| pushed_at | DATETIME | 推送时间 (UTC 存储，默认当前时间) |
| payload | TEXT | 完整的推送内容（Bark 链接） |

### 辅助表
- **Categories**: `id`, `name` (UNIQUE), `color`, `created_at`
- **Tags**: `id`, `name` (UNIQUE), `created_at`
- **Task_Tags (关联)**: `task_id`, `tag_id` (`ON DELETE CASCADE`)

## API 接口定义

| 方法 | 路径 | 描述 |
|------|------|------|
| `GET` | `/api/info` | 返回版本、状态枚举、分类、标签及系统时区 |
| `GET` | `/api/tasks` | 任务列表。支持 `q` (模糊搜索), `status`, `priority`, `category_id`, `due_date`, `remind_at`, `tag` (ID) 及 `tag_name` 筛选 |
| `POST` | `/api/tasks` | 创建任务。`tags` 字段支持传入标签名数组，自动处理创建与关联 |
| `PUT` | `/api/tasks/:id` | 修改任务。支持更新各个字段，包括 `tags` 和 `metadata` |
| `PUT` | `/api/tasks/:id/complete` | 完成任务。若有周期规则，自动推算下次时间并重置状态 |
| `DELETE` | `/api/tasks/:id` | 删除任务 |
| `GET` | `/api/categories` | 获取分类列表 |
| `POST` | `/api/categories` | 创建新分类 (支持 `name`, `color`) |
| `DELETE` | `/api/categories/:id`| 删除分类 (关联任务的 `category_id` 将被自动置空) |
| `GET` | `/api/tags` | 获取标签列表 |
| `POST` | `/api/tags` | 创建新标签 (内置正则格式校验) |
| `DELETE` | `/api/tags/:id` | 删除标签 (级联删除所有任务中该标签的关联) |
| `POST` | `/api/remind/check` | 触发提醒检查。必需参数 `?channel=cloud|agent` |
| `GET` | `/api/logs/bark` | 查询最近的 Bark 推送日志。支持 `limit` (默认 5, 最大 100) 和 `task_id` 过滤 |
