# AI 自动总结用户任务需求说明

## 1. 需求概述
通过 AI 能力对用户的任务进行周期性或手动汇总，并将总结结果生成独立、美观的网页。随后，系统会将该网页的唯一访问链接通过用户已配置的通道（如 Bark、Telegram）推送给用户，以提供优雅的长文本阅读体验。

## 2. 详细功能点

### 2.1 总结范围与数据清洗
- **数据范围**：仅包含所有“待处理 (pending)”和“处理中 (in_progress)”的任务。
- **排除项**：严格排除已完成 (completed) 和已取消 (cancelled) 的任务，确保总结聚焦于未来和当前的事项。

### 2.2 网页生成与数据存储机制
- **结构化生成**：需要为该功能专门编写一个新的 AI Prompt。AI 不直接输出 Markdown，而是被要求输出结构化且格式化的 **JSON 数据**。
- **页面模板**：使用简洁、优雅、有品味的 HTML 模板，利用 AI 吐出的 JSON 数据渲染成最终页面。该模板需在任何设备上都有优秀的展现效果。
- **唯一链接生成**：为每次总结生成一个唯一的 UUID 作为访问链接的核心路径。
- **纯数据库存储 (Cloudflare D1)**：系统将不再引入额外的文件存储（如 R2），而是直接将 AI 生成的 JSON 结果存储在数据库中。
- **状态管理与生命周期处理**：
  - **无状态设计**：数据库无需显式设计 `status` 字段，直接基于**过期时间 (`expires_at`)** 或创建时间进行动态计算和校验。
  - **访问有效期**：UUID 的公开访问有效期限定为 **24 小时**。
  - **访问校验**：当用户访问总结链接时，系统必须动态比较当前时间与 `expires_at`。只要已过期（即当前时间超过了 `expires_at`），就优雅提示用户：“页面已过期，请重新发起总结”。
  - **滚动清理机制**：后台实施滚动清理策略，定期物理删除 **30 天前**（基于 `created_at`）的历史总结数据，以防数据库膨胀。

### 2.3 消息推送 (统一流转)
- 总结完成后，将带有 UUID 的网页链接推送给用户。
- **通道策略**：所有通知通道（Bark、Telegram 等）不作差异化处理，统一推送该链接。原因是网页展示的上限更高，能统一所有通道的阅读体验。
- **Bark 专属链接格式**：需严格适配 Bark URL 跳转特性的格式，示例：`https://bark.xxxxxx.win/Goxxxxxxxxxxxx3si/URL Test?url=https://www.baidu.com`。

### 2.4 CLI 终端交互支持
- 新增终端命令：`claw-task summary`
- **交互表现**：用户手动执行该命令后，终端除了会显示最终可供点击的访问链接外，还需在终端内直接格式化展示出本次总结的核心内容，做到“所见即所得”。

### 2.5 自动化定时触发
- **触发机制**：系统支持用户自行设置定时任务来触发总结功能。
- **时区适配**：底层依赖 Cloudflare Worker 的 Cron Triggers。由于 Cron 默认基于 UTC 时间，系统的定时触发逻辑必须基于用户时区（User Timezone）配置进行精确转换和处理。

## 3. 技术实施要点

1. **AI Prompt 设计**：设计专属 System Prompt，要求大模型基于提供的未完成任务列表，进行高维度的规划与总结，并强制返回预设格式的 JSON。
2. **前端/页面模板开发**：设计一个极简、无构建依赖的高质量前端模板（HTML/CSS），用于解析渲染上述 JSON。
3. **数据库调整**：需新建数据表 `task_summaries`，字段至少包含 `id`, `uuid`, `summary_json` (存储内容), `created_at` (创建时间), 以及 `expires_at` (过期时间)。系统在访问时需依赖 `expires_at` 进行鉴权，摒弃冗余的 `status` 字段。
4. **公共路由开放**：在 API 中新增用于访问总结页面的公共路由（如 `GET /summary/:uuid`）。该路由不要求携带用户的 Bearer Token，但需验证该 UUID 的时效性（动态比较 `expires_at`）。
5. **Rust CLI 更新**：在 `cli-rust` 工程中注册新的 `summary` 命令，调用总结接口并解析返回值，实现终端 UI 的渲染。
