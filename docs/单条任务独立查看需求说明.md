# 需求说明：单条任务独立查看页面 (Task Share Page)

## 1. 背景与目标
目前系统中的任务详情主要通过 CLI 终端表格或 Web 后端管理界面查看。为了增强分享性及在移动端（如 Telegram/Bark 推送）的快速查看体验，需实现一个类似于「任务总结」的单条任务独立查看网页。该网页无需登录鉴权即可访问，但具备严格的时效性。

## 2. 核心流程
1.  **请求触发**：当用户通过 API 获取任务信息（列表或单条）时，后端自动为每个任务生成一个临时的访问凭证（UUID）。
2.  **凭证存储与优化**：将该 UUID 与任务 ID 绑定，并设置 24 小时后的过期时间，存入数据库。若任务已有未过期的凭证，则复用现有凭证以降低数据库写入压力。
3.  **字段返回**：在任务对象的 JSON 响应中新增 `view_url` 字段，格式为 `https://<domain>/share/:uuid`。
4.  **页面访问**：用户点击链接，系统校验 UUID 是否存在且未过期。校验通过后，从 `tasks` 表读取实时数据并渲染为精美的 HTML 页面。

## 3. 详细设计规格

### 3.1 数据库调整
新建数据表 `task_shares`：
*   `uuid` (TEXT, PRIMARY KEY): 随机生成的唯一标识符。
*   `task_id` (INTEGER): 关联的任务 ID。
*   `expires_at` (DATETIME): 过期时间（生成时间 + 24小时）。
*   **索引优化**：在 `task_id` 和 `expires_at` 上建立索引，分别优化“查重/生成”和“过期清理”逻辑。

### 3.2 后端接口变更
*   **公共路由**：`GET /share/:uuid`
    *   **渲染**：使用原生 HTML/CSS 渲染。UI 风格与 `summary` 页面保持一致。
*   **逻辑层**：
    *   `src/worker/handlers/share.ts`: 实现 `getOrCreateShareUrls` (批量) 和 `createShareUrl` (单条) 辅助函数。
    *   `src/worker/handlers/tasks.ts`: 在 `GET/POST/PUT` 响应逻辑中注入 `view_url`。

### 3.3 CLI 展示策略 (AI 友好性决策)
基于 **AI 优先 (AI-First)** 的原则，对 CLI 的展示逻辑进行了精细化区分：
*   **列表视图 (`list`)**：表格中**不显示** `view_url` 列。
    *   *决策逻辑*：长 URL 会破坏表格结构，且对 AI 代理（Agent）来说是高 Token 成本的噪声。
*   **单条反馈 (`add`, `update`, `ai`)**：在操作成功的反馈信息中显式打印 `View Online: <url>`。
    *   *决策逻辑*：方便人类用户即刻查阅，同时也为 AI 提供了转发给人类时的高价值媒介。
*   **单条筛选参数**：为 `list` 命令增加 `-i` / `--id` 参数，支持通过 ID 快速获取单条任务的简洁表格视图（不含 URL），确保 AI 处理时的完备性与解析友好。

### 3.4 UI 交互设计
*   **响应式布局**：优化移动端阅读体验，适配 Telegram 内部浏览器。
*   **内容完备**：展示任务全字段，包括描述（支持换行）、优先级（颜色区分）、元数据（AI 溯源上下文）等。

## 4. 安全与性能考量
*   **时效性管理**：链接 24 小时自动失效。系统通过 Cron 定期清理过期记录。
*   **只读权限**：`/share` 路由不具备任何写操作权限，确保公开访问的数据安全。
