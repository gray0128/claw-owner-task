# 聊天渠道内容展示逻辑总结 (Chatbots Content Display Logic)

本文档总结了系统中针对聊天类渠道（如 Telegram 等）在内容展示和交互处理方面的核心逻辑。为方便后续扩展至其他 IM 平台（如 WeChat、QQ 等），将内容划分为“通用逻辑”与特定平台的“专属逻辑”。

## 1. 通用逻辑（适用于所有聊天渠道）

在设计聊天机器人的交互时，系统遵循以下通用的展示与流转原则，以平衡安全性、响应速度与用户体验：

### 1.1 友好的鉴权与引导反馈
*   **白名单拦截**：系统基于来源的身份标识（如 User ID / Chat ID）进行严格的白名单鉴权，只允许预授权的用户与机器人进行交互。
*   **拒绝并告知 (Reject with Info)**：当未授权用户触发交互时，系统不会采取“静默失败”策略，而是会回复一条明确的拒绝信息，并在信息中主动提供该用户的真实 ID（例如：*“鉴权失败，你的 ID 是 xxx，请将其加入配置”*）。这极大地降低了个人用户的部署与调试门槛。

### 1.2 意图分流：快捷指令 vs 自然语言
为了兼顾响应的确定性和交互的灵活性，用户的输入会基于特定前缀（通常是 `/`）进行分流：
*   **快捷指令（直达业务层）**：对于带有 `/` 前缀的明确指令（如 `/add`, `/summary`），系统直接将其路由至后台对应的 API Handler，**完全绕过 AI 处理层**。这种机制确保了极高的准确率和极低的响应延迟。
*   **自然语言（AI 代理层）**：无特殊前缀的普通对话文本，则会完整传入 AI 代理模块，由大模型进行语义理解、意图提取并映射为系统后端的标准动作（Create, Update, Query, Complete 等）。

### 1.3 异步交互与体验优化
*   **即时占位反馈 (Loading State)**：针对可能耗时较长或依赖外部调用的操作（例如 `/summary` 需要全量检索未完成任务并交由 AI 生成长文本），系统会在接收到指令后立即返回一条状态消息（如：*“⏳ 正在生成任务总结，请稍候...”*），有效缓解用户的等待焦虑。
*   **参数缺失引导**：当用户触发快捷指令但未提供必要的内容参数时，系统会返回包含格式示例的提示化引导（如：*“ℹ️ 请提供任务内容，例如：/add 买牛奶”*）。

### 1.4 结构化的 AI 结果回显
无论底层 AI 将自然语言解析成了什么具体动作，系统最终都会将后台返回的 JSON 数据转化为具有一致性排版和良好可读性的聊天气泡：
*   **结果定性锚点**：消息首行固定使用 `✅ 操作成功` 或 `❌ 操作失败` 配合 Emoji 作为明显的视觉锚点。
*   **意图透明化 (Intent Echo)**：在操作成功时，系统会在消息中显式展示 AI 最终解析出的动作类型（如：*“解析动作: create_task”*）。这增加了系统的透明度，让用户直观地感知当前 AI 对自己意图的理解程度。
*   **动态数据适配展现**：
    *   **列表数据（多条）**：提示“找到 X 个任务”；逐条列出任务信息，突出**标题（加粗）**与**状态（如 [已完成]）**，并在新行附加 📅 截止时间或 ⏰ 提醒时间。
    *   **单体数据（单条/增改）**：结构化展示任务的标题、唯一的 Task ID，以及相关的日期字段。
    *   **异常拦截（失败）**：将底层的报错日志以等宽代码块（Code Block）的形式呈现，以便用户快速识别是格式错误还是系统错误。

## 2. Telegram 渠道专属逻辑

受限于 Telegram Bot API 的特殊限制及展示机制，项目中实现了以下针对 Telegram 的底层处理策略：

### 2.1 专属的 Parse Mode (HTML 渲染引擎)
*   **弃用 Markdown 引擎**：尽管 Markdown (尤其是 MarkdownV2) 在 Telegram 中受支持，但其对标点符号（如 `.`, `-`, `_`, `(` 等）的转义要求过于严苛。用户通过自然语言生成的任务描述中极易包含这些符号，往往会导致整个 JSON Payload 被 Telegram API 拒绝。
*   **强制 HTML 模式**：系统全局强制指定 `parse_mode: 'HTML'`。在构建气泡时仅使用经过验证的语义化标签：
    *   `<b>`: 用于加粗强调核心数据（如结果状态、任务标题）。
    *   `<i>`: 用于斜体展示说明性内容（如 AI 意图回显）。
    *   `<pre>`: 用于等宽渲染长段的报错日志。

### 2.2 严格的安全防护与 HTML 转义机制
*   **闭合标签防崩溃**：因为采用 HTML 模式，如果用户输入的任务内容中包含 `<` 或 `>` 符号（例如：`买 3 < 4 个苹果`），Telegram 会尝试将其解析为 HTML 标签，导致解析失败从而丢弃整条消息。
*   **全量内容转义**：为了解决上述问题，系统在拼接 Telegram 消息体之前，**强制**让所有可能包含动态用户输入的内容经过 `escapeTelegramHTML` 函数处理，将 `&`, `<`, `>` 安全替换为对应的 HTML 实体（`&amp;`, `&lt;`, `&gt;`）。

### 2.3 Webhook 环境的特殊处理
*   **配置防呆校验**：在比对白名单鉴权（判断环境变量 `TELEGRAM_CHAT_ID` 与请求体中的 Chat ID 是否一致）时，由于从环境中读取的值极易混入不可见的空格，系统在对比前强制对双方的 ID 执行了 `.trim()` 操作。
*   **特定层级解析**：不同于一般的标准化 API，处理逻辑强耦合于 Telegram Webhook 下发的特有结构（严格从 `body.message.text` 提取输入，从 `body.message.chat.id` 提取用户身份）。
